Introduction to WebAssembly

WebAssembly (abbreviated Wasm) is a binary instruction format for a stack-based virtual machine. Wasm is designed as a portable target for compilation of high-level languages like C/C++/Rust, enabling deployment on the web for client and server applications.

- webassembly.org

Memory in WASM

  • Linear memory - single contiguous block of bytes

    • represented by Javascript ArrayBuffer in browser

  • Sandboxed, every instance is isolated / exclusive

    • ​ie. Code & call stack not accessible by other WASM instances

Glue code

  • WASM only supports numeric types: i32, i64, f32, f64

  • Conduct pointer arithmetic to access other types

Existing tooling

  • https://github.com/rustwasm/wasm-pack
  • https://github.com/emscripten-core/emscripten
  • Such tools generate the glue code (bindings) for you
    • In the form of a .js file
    • ie. you call the .js file instead of the wasm module

  • No access to DOM

Limitations

  • Module size
    • everything is loaded in memory
    • shipping 50MB over network is not ideal
  • Threading - limited support
    • SharedArrayBuffer - had to be disabled because of Spectre (security vulnerabilities)
  • Garbage collection (GC) - not available in MVP 
  • Debugging across languages
    • Write code in one language which errors in another
    • What kinds of tooling are available?
  • Security concerns
    • harder to inspect binary files

What happens to JS?

Best of both worlds

  • Leverage JS ecosystem
    • JS has libraries for DOM manipulation (eg. Vuejs, React, etc.) 
  • Use WASM for computation
  • Augment each other, not competitors

Should I use WASM?

  • Not supported on IE
  • Do you have a need for it?
    • I/O vs CPU bound
      • Remember interfacing with DOM is slow
      • Loading WASM module is an overhead
    • Profile & identify bottlenecks
  • Community support
    • Newer technology = uncharted territories
